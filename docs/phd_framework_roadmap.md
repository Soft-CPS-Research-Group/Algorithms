# REC Flexibility Control Framework â€” Architectural Blueprint (Revised)

## Purpose of this document

This document describes, in implementation-oriented detail, a proposed AI control architecture for Renewable Energy Communities (RECs), designed around four core thesis goals:

- Deployability
- Safety
- Robustness
- Scalability

The intent is to serve as a blueprint for coding and iteration, especially with Codex or any engineering workflow that will gradually implement and refine each subsystem.

This version reflects two important design clarifications:

1. Safety is not treated as a third hierarchical layer.
   It is modeled as a cross-cutting execution and safety subsystem that sits below and around the learned controllers.
2. The community controller representation is intentionally left open.
   It is still unclear whether a set-based encoder is truly needed at that level, or whether a more stable aggregate/state-vector design is better.

## 1. High-level architecture

The recommended architecture has:

### Hierarchical control levels

- Level 1 â€” Community Coordinator
- Level 2 â€” Universal Local Controller

### Cross-cutting subsystem

- Safety and Execution Subsystem

So the system is best described as:

> a 2-level hierarchical controller with a cross-cutting safety and execution subsystem

This avoids the misleading impression that there are three HRL layers.

---

## 2. Core design philosophy

### 2.1 Two-level hierarchy

Different decisions belong to different abstraction levels:

- Community level decides what should be achieved globally.
- Building level decides how to achieve it locally.

### 2.2 Safety is not another HRL level

Safety does not define another abstraction level of control intent.

Instead, it guarantees that proposed actions generated by learned controllers are:

- feasible
- safe
- electrically valid
- robust enough for execution

So safety is better thought of as:

- an execution guard
- a feasibility corrector
- a deployment-enabling subsystem

### 2.3 Variable structure support

The architecture must support:

- different numbers of buildings in the REC
- buildings entering/leaving the REC
- buildings with different assets
- buildings with no controllable assets
- buildings with many controllable assets
- mono-phase and tri-phase constraints
- future addition/removal of device types

### 2.4 Temporal continuity

Controllers should not behave as purely memoryless step-to-step policies.

They should operate in closed loop, every time step, but using:

- internal memory
- short-horizon consistency
- receding horizon target logic
- action smoothness

### 2.5 Safe-by-design execution

Safety should not exist only as a reward penalty.

It should exist as:

- explicit constraints
- action filtering/projection
- fallback behavior
- uncertainty-aware execution where needed

## 3. Temporal resolution and control cycle

### Recommended base resolution

A good default is:

- Decision step: 15 minutes

This aligns with:

- electricity pricing granularity
- solar and load forecast granularity
- realistic EMS control horizons
- battery and EV control timescales

### Receding-horizon logic

The controller can operate every 15 minutes, but should reason over a short future horizon.

Suggested horizons:

- Level 1 â€” Community Coordinator: 4 to 24 steps ahead (1 to 6 hours)
- Level 2 â€” Local Controller: 2 to 8 steps ahead (30 minutes to 2 hours)
- Safety/Execution subsystem: current-step feasibility with optional short-horizon correction

### Closed-loop cycle

At each 15-minute step:

1. Read latest observations.
2. Update forecasts/context.
3. Community Coordinator generates community-to-building targets.
4. Local Controller generates proposed per-asset actions.
5. Safety/Execution subsystem checks, filters, corrects, executes.
6. System transitions.
7. Repeat.

## 4. Level 1 â€” Community Coordinator

### 4.1 Purpose

The Community Coordinator is responsible for global coordination.

It should reason about:

- aggregated demand and production
- community import/export strategy
- use of available flexibility
- fairness/equitable coordination
- peak shaving
- self-consumption
- energy sharing incentives
- near-future event preparation

It says:

> What should each building roughly try to do over the next period?

not:

> Set charger 3 to 4.2 kW right now.

### 4.2 Inputs the Community Coordinator should see

This level should primarily see building-level summaries, not raw asset-level detail.

#### Global inputs

- time features
- tariff / price now and ahead
- local market / sharing discount signal
- weather summary
- solar forecast
- aggregate load forecast
- aggregate PV forecast
- aggregate import/export history
- recent target-tracking error
- recent flexibility usage history

#### Per-building summary inputs

- building ID embedding (or anonymous slot embedding)
- building type/category
- baseline load estimate
- forecasted local net demand
- current import/export
- available upward/downward flexibility
- available EV flexibility
- available battery flexibility
- expected controllable power envelope
- comfort/priority weight (if needed)
- target compliance history
- uncertainty summary

### 4.3 Open design doubt (keep explicit)

Open question:

- Does the Community Coordinator need building-wise set encoding?
- Or is a stable aggregate/global state enough?

Why this matters:

- if the real role is mostly aggregate coordination, set encoding may be unnecessary complexity

Possible paths:

- Path A â€” Aggregate-first coordinator
- Path B â€” Hybrid aggregate + compact building summaries
- Path C â€” Building-aware set encoder

Current recommendation:

- do not hard-freeze this decision yet
- keep this as a deliberate experimental branch

### 4.4 Candidate technologies

#### Option A â€” Stable aggregate vector model

- Inputs: aggregate REC state + forecasts + flexibility summaries
- Model: MLP + GRU
- Algorithm: PPO (or other stable policy gradient)

#### Option B â€” Hybrid aggregate + compact summaries

- Inputs: aggregate state + limited per-building summaries
- Model: aggregate encoder + summary pooling + GRU
- Algorithm: PPO

#### Option C â€” Set-based building-aware

- Inputs: variable set of building summaries + global context
- Model: Deep Sets / Set Transformer / attention pooling + GRU
- Algorithm: PPO

Design note:

- Option A or B may be better aligned with the intended role than Option C in early phases.

### 4.5 Outputs

This level should output targets, not device commands.

Possible outputs:

- per-building target net import/export
- per-building flexibility budget (up/down, storage, EV)
- per-building priority/urgency/fairness signal

Practical recommendation:

- target net power + flexibility budget + optional confidence/urgency scalar

### 4.6 Temporal continuity

Recommended behavior:

- output current target
- reason over future steps internally
- optionally output short target envelope
- replan every step

### 4.7 Community constraints

Coordinator should respect high-level limits:

- community import cap
- export limits
- fairness bounds
- sharing rules
- aggregate feasibility assumptions

Mechanisms:

- clipping
- soft constraints
- explicit projection
- constrained RL (later)

### 4.8 Training strategy

- train in simulation across synthetic RECs
- vary building count/types/churn
- vary forecasts/prices/participation
- vary asset availability summaries

Main goal:

- generalize across communities without full retraining

Offline pretraining:

- optional light use for initialization and stabilization

### 4.9 Future upgrades

- constrained PPO/CMDP-style variants
- fairness-aware objectives
- uncertainty-aware target envelopes
- limited world-model target planning
- event-conditioned scheduling
- policy distillation for interpretability

## 5. Level 2 â€” Universal Local Controller

### 5.1 Purpose

The Local Controller is the core execution intelligence at building level.

It receives:

- local state
- local forecasts
- community target/budget

and outputs:

- proposed per-asset actions

The key challenge is topology heterogeneity and variability:

- buildings with/without controllable assets
- changing asset sets over time

### 5.2 Main requirement

This must be a universal local controller.

It should not require:

- architecture per building
- fixed asset count
- full retraining when assets change

It should operate on:

- building-level context
- variable set of asset descriptors (tokens)

### 5.3 Local input structure

#### Group A â€” Building context

- time features
- tariff / intra-community energy cost
- community target
- current net meter power
- local forecast load/PV
- recent tracking error
- local preferences
- phase configuration summary
- uncertainty summary

#### Group B â€” Asset tokens

For each controllable/relevant asset:

- asset type embedding
- current power
- controllable power bounds
- SoC (if applicable)
- energy capacity (if applicable)
- connected/availability flag
- deadline information
- desired energy/charge requirement
- phase assignment
- efficiency/loss factors
- health/reliability flags

#### Group C â€” Historical context

Recent window of:

- meter power
- asset states
- targets received
- actions taken
- constraint activation history
- forecast error indicators

#### Group D â€” Optional exogenous context

- weather
- occupancy proxy
- forecast confidence
- abnormal-event flags

### 5.4 Candidate AI technologies

#### Asset representation

Recommended default:

- shared asset encoder
- asset type embeddings
- set aggregation/attention
- masking for absent assets

Options:

- Deep Sets
- Set Transformer
- masked attention encoder
- tokenized asset representation

#### Temporal modeling

- first choice: GRU
- later: sequence transformer / memory-augmented model

#### Output decoding

- shared per-asset decoder
- parallel decoding over asset tokens
- masked outputs
- optional autoregressive decoding later

#### RL algorithm candidates

- PPO with custom architecture
- SAC with custom architecture
- TD3 when smooth continuous control is critical

Default stance:

- avoid local MARL as default formulation
- treat local control as one structured policy over assets

### 5.5 Why not local MADDPG as default

MADDPG-style local decomposition becomes fragile under:

- variable asset count
- new device types
- strong inter-building heterogeneity
- deployment demands for genericity

A universal local policy with variable asset support aligns better with thesis goals.

### 5.6 Outputs

- EV charger power
- battery charge/discharge power
- PV curtailment
- flexible load activation/shift
- reserve holding decisions

Optional confidence outputs:

- action confidence
- expected feasibility score

### 5.7 Temporal continuity

Required to avoid:

- oscillation
- unnecessary switching
- unstable battery usage
- poor EV deadline handling

Mechanisms:

- recurrent state (GRU/LSTM)
- previous action/target features
- action smoothness/ramp penalties
- switching penalties
- optional short local plan

### 5.8 Current action vs short plan

Recommended:

- always output current executable action
- optionally infer a short plan internally
- execute only first step
- replan every step

### 5.9 Offline pretraining opportunity

Strong recommendation:

- concentrate offline pretraining efforts in the Local Controller

Suggested stages:

1. Archetype datasets
   - no controllable assets
   - PV only
   - EV only
   - BESS only
   - EV+PV
   - EV+PV+BESS
   - office/multi-charger
2. Universal mixed pretraining
3. Online safe fine-tuning with low exploration budget

### 5.10 High-impact innovation options

- universal sequence policy over temporal + asset tokens
- set-based universal controller with GRU and per-asset decoder
- adapter-based specialization (lightweight adaptation)

### 5.11 Future upgrades

- uncertainty-aware local control
- risk-sensitive objectives
- sequence transformer migration
- meta-learning for rapid adaptation
- world-model-assisted planning
- interpretable policy distillation
- compositional new-asset plug-in modules

## 6. Safety and Execution Subsystem

### 6.1 Purpose

Not an HRL level.

Cross-cutting subsystem between proposed actions and physical execution.

Ensures actions are:

- feasible
- safe
- electrically valid
- contractually/operationally compliant
- robust under uncertainty where possible

### 6.2 Responsibilities

- power bounds
- SoC limits
- connection validity
- ramping constraints
- minimum on/off constraints
- phase limits and mono/tri-phase consistency
- anti-simultaneous charge/discharge
- deadline feasibility checks
- import/export caps
- critical fallback modes

### 6.3 Mechanism families

A. Deterministic rule filters

- clip to range
- disallow disconnected actions
- anti-conflicting commands
- enforce phase limits

B. Action projection

- project action vector to feasible set

C. Optimization-based corrector

- nearest safe action via local optimization

D. Fallback controller

- RBC, MPC, or emergency heuristic

### 6.4 Inputs

- proposed actions
- real-time asset states
- device availability
- SoC/power/phase data
- rule configuration
- operating-mode flags
- safety margins
- uncertainty estimates
- target criticality

### 6.5 Outputs

- safe executable actions
- correction reason codes
- safety flags
- fallback indicator
- violation counters
- uncertainty override indicator

### 6.6 Temporal continuity

- ramp-rate enforcement
- minimum dwell time
- anti-oscillation logic
- phase-balance continuity heuristics

### 6.7 Advanced safety upgrades

- uncertainty-aware safety margins
- conformal prediction bands
- predictive safety filters
- learned shields

### 6.8 Implementation path

Start with:

- deterministic checks
- clipping
- projection
- fallback RBC

Then evolve to:

- optimization-based correction
- uncertainty-aware margins
- predictive safety filters

## 7. Cross-subsystem interfaces

### 7.1 Community Coordinator -> Local Controller

- target net import/export
- up/down flexibility budget
- urgency/priority scalar
- optional short-horizon envelope
- optional fairness weight
- optional confidence

### 7.2 Local Controller -> Safety/Execution

- proposed per-asset action
- optional confidence/uncertainty
- optional expected feasibility score
- optional short-plan summary

### 7.3 Feedback upward

Safety and local subsystems should report:

- target tracking quality
- constraint saturation
- flexibility exhaustion
- fallback frequency
- uncertainty level
- asset availability degradation

## 8. Training and experimentation strategy

### 8.1 Implementation order

Phase 1:

- architecture skeleton
- deterministic safety/execution

Phase 2:

- stronger variable-asset encoding
- recurrent improvements
- smoothness penalties
- stronger offline local pretraining

Phase 3:

- thesis-grade innovations
- uncertainty-aware safety
- optional world-model support

### 8.2 Baseline ladder

- RBC
- MPC (if feasible)
- current MADDPG
- hierarchical without universal local controller
- hierarchical without safety subsystem
- hierarchical without recurrence
- hierarchical with fixed local vector controller
- final architecture

### 8.3 Evaluation axes

Performance:

- cost
- self-consumption
- peak shaving
- flexibility use
- target tracking

Safety:

- violation count/severity
- fallback frequency
- unsafe-proposal rate

Robustness:

- noise/missing data/delays
- forecast errors
- asset churn
- REC membership churn

Scalability:

- buildings count
- assets per building
- inference latency
- retraining burden

Deployability:

- adaptation effort
- retraining needs
- correction explainability
- degradation under real-world imperfections

## 11. Minimal viable version vs thesis version

### 11.1 Minimal viable version

Community Coordinator:

- aggregate or hybrid encoder
- GRU
- PPO
- single-step target (+ optional short envelope)

Local Controller:

- asset token builder
- Deep Sets or simple attention
- GRU
- per-asset continuous decoder

Safety/Execution:

- clipping
- feasibility rules
- anti-oscillation heuristics
- fallback RBC

### 11.2 Strong thesis version

Community Coordinator:

- recurrent coordinator
- constrained target generation
- uncertainty-aware envelopes
- representation selected by ablation evidence

Local Controller:

- universal sequence policy
- token-based local representation
- offline archetype pretraining
- fast adaptation via adapters / low-rank updates

Safety/Execution:

- projection-based safety
- uncertainty-aware margins
- predictive safety filter
- interpretable correction logs

## 12. Final recommendation

Stable choices to adopt now:

- 2-level hierarchy
- 15-minute control step
- receding-horizon reasoning
- recurrent memory
- universal local controller
- no local MARL by default
- explicit safety and execution subsystem
- offline pretraining focused on local controller

Choices to postpone until after first implementation:

- final Community Coordinator representation
- transformer migration timing
- conformal safety
- world-model planning
- adapter-based specialization
- uncertainty-aware constrained RL variants

> A deployable hierarchical AI control framework for Renewable Energy Communities, combining recurrent community coordination, universal local control over variable asset sets, and explicit safe execution under physical and operational constraints.
