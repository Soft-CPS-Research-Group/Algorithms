# Offline RL Dataset — Field Reference

Documentation for the EV charging offline RL dataset generated by `scripts/generate_offline_dataset.py`.

---

## Structural Fields

### `episode_id`
- **Example**: `5`
- **Type**: Integer
- **Description**: A unique integer identifying each charging session. One episode = one EV plugging in, charging, and departing. Assigned sequentially as sessions are extracted across all chargers.

### `timestep`
- **Example**: `0`, `7`, `14`
- **Type**: Integer
- **Description**: The step index *within* the episode, starting at 0. Not a global clock — it's relative to when this session started. Each timestep represents **1 hour** (from `schema.json`: `seconds_per_time_step: 3600`).

### `charger_id`
- **Example**: `charger_1_1`
- **Type**: String
- **Description**: Which physical charger this row belongs to. The naming convention is `charger_{building}_{index}` — so `charger_1_1` is the first charger at Building 1. Parsed from the charger CSV filename and matched to specs in `schema.json`.

### `ev_id`
- **Example**: `Electric_Vehicle_7`
- **Type**: String
- **Description**: Which EV is connected during this session. Taken directly from the `electric_vehicle_id` column in the charger CSV. This determines the battery capacity lookup from `schema.json` (e.g. EV_7 → 60 kWh).

## State Vector (observation at time *t*)

### `hour`
- **Example**: `17`
- **Range**: 0–24
- **Source**: Building CSV (`Building_X.csv`)
- **Description**: Hour of the day. This is the global clock — hour 17 = 5 PM.

### `day_type`
- **Example**: `5`
- **Range**: 1–7
- **Source**: Building CSV
- **Description**: Day of the week (1=Monday … 7=Sunday). Day 5 = Friday. Useful for the RL agent to learn weekday/weekend patterns.

### `soc`
- **Example**: `78.0`
- **Range**: [0, 100] (percentage)
- **Source**: **Simulated** (initial value from charger CSV, then forward-simulated)
- **Description**: The EV's current State of Charge in percentage. For the first step of an episode, it comes from `electric_vehicle_estimated_soc_arrival` in the charger CSV (the SoC when the car arrived). For subsequent steps, it equals the `next_soc` from the previous step — i.e., it's simulated forward using:

```
soc += (action × max_power × efficiency × 1h) / capacity × 100
```

Example: 78% initial → after one step at action=0.6: `78 + (0.6 × 11kW × 0.95 × 1h) / 60kWh × 100 = 88.45%`.

### `required_soc`
- **Example**: `87.0`
- **Range**: [60, 100] (percentage)
- **Source**: Charger CSV (`electric_vehicle_required_soc_departure`)
- **Description**: The target SoC the EV needs when it departs. Stays constant throughout the session. This is what the agent is trying to achieve.

### `time_to_departure`
- **Example**: `14.0`
- **Range**: [0, 23] (hours)
- **Source**: Charger CSV (`electric_vehicle_departure_time`)
- **Description**: Hours remaining until the EV needs to leave. Acts as a countdown (decrements by 1 each hour). At 0.0 the car is about to depart. This is the urgency signal for the agent.

### `solar_generation`
- **Example**: `246.525`
- **Range**: [0, ~958] (Wh)
- **Source**: Building CSV (`solar_generation`)
- **Description**: PV solar generation at the building for this hour, in Wh. At hour 17 (5 PM) there's still daylight, hence 246 Wh. At night it drops to 0. The RBC uses this to decide when to charge preferentially.

### `electricity_pricing`
- **Example**: `0.124`
- **Range**: [0, ~0.65]
- **Source**: `pricing.csv`
- **Description**: Electricity price at this hour ($/kWh or equivalent unit). A community-level feature shared across all buildings. Lower values = cheaper to charge from the grid.

### `carbon_intensity`
- **Example**: `0.197596`
- **Range**: [~0.07, ~0.28]
- **Source**: `carbon_intensity.csv`
- **Description**: Grid carbon intensity at this hour (kgCO₂/kWh or similar). Not directly used in the reward function but included as a state feature — an offline RL agent could learn to minimize carbon.

### `max_charging_power`
- **Example**: `11.0`
- **Range**: {7.4, 11.0, 22.0} (kW)
- **Source**: `schema.json` → building → charger → `max_charging_power`
- **Description**: The charger's maximum power in kW. Static per charger (11 kW for charger_1_1, 22 kW for charger_4_1, etc.). Included so the RL agent knows the charger's capability — essential when combining chargers with different specs.

### `battery_capacity`
- **Example**: `60.0`
- **Range**: [40, 100] (kWh)
- **Source**: `schema.json` → `electric_vehicles_def` → EV name → `battery.attributes.capacity`
- **Description**: The connected EV's battery capacity in kWh. Static per episode (same car throughout). Ranges from 40–100 kWh across the 8 EV types.

## Action & Reward

### `action`
- **Example**: `0.6`
- **Range**: [0, 1]
- **Source**: Computed by the RBC (Rule-Based Controller)
- **Description**: The normalized charge rate. Represents the fraction of `max_charging_power` to apply. So action=0.6 on an 11 kW charger means charging at 6.6 kW.

**How the RBC decides:**
1. Calculates `energy_needed = (required_soc − current_soc) / 100 × capacity`
2. Computes minimum charge rate: `required_power = energy_needed / time_to_departure`
3. Normalizes: `action = min(1.0, required_power / max_power)`
4. **Solar boost**: if solar > 0 → bumps action to at least 0.6 (the `pv_preferred_charge_rate`)
5. **Emergency**: if time_to_departure ≤ 1h → forces action to 1.0
6. **Flexibility**: if lots of time and no solar → trickle-charges (action → 0)

### `reward`
- **Example**: `0.618716`
- **Range**: [~−0.72, ~7.80]
- **Source**: Computed by the reward function
- **Description**: The per-step reward signal, computed as:

```
r = 0.5 × r_progress + 0.2 × r_solar − 0.1 × r_cost + r_terminal
```

**Component breakdown (first step example — soc: 78→88.45, solar=246.5, price=0.124, action=0.6):**

| Component | Formula | Value |
|-----------|---------|-------|
| **r_progress** | `0.5 × (88.45 − 78) / (87 − 78)` | 0.58 |
| **r_solar** | `0.2 × min(1, 246.5 / (0.6×11+ε)) × 0.6` | 0.12 |
| **r_cost** | `0.1 × 0.124 × 0.6 × 11` | −0.082 |
| **r_terminal** | 0 (not last step) | 0.0 |
| **Total** | | **≈ 0.618** |

At the terminal step: SoC=88.45 ≥ 87−1 → `r_terminal = +1.0`. Action=0 so no other components → reward = **1.0** exactly.

**Reward component details:**

- **r_progress** — Dense per-step signal rewarding progress toward the departure SoC target. Normalised by the initial gap so that each unit of progress contributes equally regardless of absolute gap size.
- **r_solar** — Bonus proportional to how much of the charged energy could be covered by local PV generation. Encourages the agent to shift charging into high-solar periods.
- **r_cost** — Penalty for the monetary cost of drawing energy from the grid. Encourages off-peak / low-price charging.
- **r_terminal** — Only at the last step of an episode: `+1.0` if final SoC ≥ required SoC − 1%, else `−0.01 × shortfall`. Provides a large end-of-episode signal for meeting / missing the departure target.

## Next State (*s'* — observation at time *t+1*)

These are the state features at the **next timestep**, enabling (s, a, r, s') tuples without needing to look ahead in the CSV.

### `next_soc`
- **Example**: `88.45`
- **Description**: SoC after the action is applied. Computed by the SoC simulation formula. This becomes `soc` in the next row.

### `next_hour`, `next_day_type`, `next_solar_generation`, `next_electricity_pricing`, `next_carbon_intensity`
- **Description**: The corresponding state features at the next timestep. Read from the next row's building/pricing/carbon data. Note how at the day boundary: hour goes 24→1 and day_type changes (e.g. 5=Fri → 6=Sat).

### `next_time_to_departure`
- **Example**: `13.0`
- **Description**: The countdown at the next step (decrements by 1 each hour). Read from the next charger CSV row.

> For the **terminal step** (done=1), these "next" values are copied from the current step since there is no true next state.

---

## Terminal Flag

### `done`
- **Example**: `0` or `1`
- **Type**: Integer (binary)
- **Description**: `1` at the last step of an episode (EV is about to depart), `0` otherwise. Determined by whether this is the last `charger_state=1` row before the charger goes to state 3 (no car). Exactly 2,788 rows have done=1, matching the 2,788 episodes.

## Additional Dataset Information

- **Data source alignment**: Building CSVs have 8,760 rows (1 year, hourly), charger CSVs have 9,527 rows. The script truncates to 8,760 to keep everything aligned.
- **First-episode fallback**: The very first session in each charger file starts with state=1 (car already connected, no arrival event). Since there's no arrival SoC, a default of 50% is used. All subsequent sessions get their true arrival SoC from the state=2 row.
- **Sessions skipped**: 7 sessions with only 1 step were discarded (too short for meaningful transitions).
- **Companion file**: `ev_charging_all.meta.json` stores the exact RBC hyperparameters, reward weights, and statistics used — so the dataset is fully reproducible.
