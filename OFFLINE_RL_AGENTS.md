# Offline RL — Session Bootstrap (v2)

> **Read me first** in any new conversation about offline RL on this repo.
> This file is the single source of truth for *what* we are doing, *why* we
> restarted, and *how* to pick up where we left off.

---

## 1. Goal in one paragraph

Train an offline reinforcement-learning agent on **Building 5** of the
CityLearn `citylearn_three_phase_dynamic_topology_demo_v1` dataset that
beats the rule-based controller (RBC) on the official CityLearn KPIs
(cost, carbon, peak, ramping, load factor, unserved energy). The agent
must learn from a **fixed dataset of (obs, action, reward, next_obs)
transitions** generated by a behaviour policy — no online interaction
during training. We start with the RBC as the behaviour policy, then
iterate by swapping it for stronger policies (BC → IQL → …).

---

## 2. Glossary (plain-English)

| Term | Meaning |
|------|---------|
| **RBC** | Rule-Based Controller. Hand-crafted heuristic. Our starting behaviour policy. |
| **Behaviour policy** | The policy that *generated* the offline dataset. |
| **Offline RL** | Training a policy from a frozen dataset, no live env. |
| **BC** | Behaviour Cloning. Supervised regression `state → action`. Reproduces the behaviour policy, cannot exceed it. |
| **IQL** | Implicit Q-Learning. Offline RL algorithm that *can* exceed the behaviour policy by re-weighting the dataset toward higher-return actions. |
| **Transition** | One row: `(obs_t, action_t, reward_t, obs_{t+1}, done_t)`. |
| **Episode** | A full simulator rollout (typically 1 year = 8760 hourly steps). |
| **KPI** | CityLearn evaluation metric. Normalised so `1.0` = no-control baseline; **lower is better**. |
| **V2G** | Vehicle-to-Grid. Bidirectional EV charging. |

## 3. What v1 was, and what went wrong

v1 ran across milestones M1–M4 under `datasets/offline_rl/` and
`runs/offline_*`. v1 is **frozen** and lives untouched as historical
reference.

- **M2** collected an offline dataset for a single agent (Building 5).
- **M3** trained a BC baseline. It "matched" RBC closely on KPIs.
- **M4** trained IQL with a per-building reward proxy. IQL also "matched"
  RBC closely.

What we discovered at the end of M4:

1. The dataset's `action_clean_*` columns (the supervision targets used
   by BC and IQL) are **all zeros** for Building 5.
2. BC and IQL both learned the constant function `tanh(0) = 0`.
3. Their "parity with RBC" was an artefact of doing nothing — the env
   itself is mild enough that doing nothing scores close to RBC on KPIs.
4. The reward used in M4 (`reward_b5_raw = -(grid_cost + λ_carbon·carbon
   + λ_unserved·unserved)`) covers cost, carbon and unserved energy but
   **omits peak, ramping and load-factor** — three KPIs the agent's
   actions directly influence. So even with a non-degenerate dataset,
   the reward could not have driven improvement on those axes.
5. **(Discovered during v2 step 2.)** The v1 `RuleBasedPolicy` (still
   present, untouched, in `algorithms/agents/rbc_agent.py`) reads EV
   observations using bare names like `electric_vehicle_charger_state`,
   but the env exposes those fields **namespaced by charger ID**
   (e.g. `electric_vehicle_charger_charger_5_1_connected_state`). Every
   lookup defaulted, `charger_state` returned 0, and the policy
   short-circuited to `return 0.0` on every step. **The "RBC" baseline
   in every v1 benchmark was a do-nothing policy.** The "tied with RBC"
   results in M3/M4 reflect BC and IQL also doing nothing, not
   imitating RBC correctly. v2 wraps the RBC in
   `algorithms/offline_rl_v2/rbc_v2.py` to fix the obs-name resolution
   without touching v1.
6. **(Discovered during v2 step 3.)** With 10 RBC seeds, NNLS calibration
   produced multicollinear weights — `cost` and `carbon` got
   non-zero weights but `peak` and `ramp` were zeroed because all four
   term sums move together across seeds. v2 deviates from
   `reward_design_v2.md` §4.1 with an explicit **hybrid floor rule**:
   NNLS-positive weights are kept; NNLS-zero / skipped terms fall back
   to the §4.2 defaults expressed in standardised space
   (`w_raw[k] = DEFAULT_WEIGHTS[k] / σ_k`). The §4.2 ratios are
   preserved; the magnitudes are on a comparable per-σ scale. Per-term
   provenance is recorded in `reward_v2_weights.json :: weight_source`.
   See `docs/offline_rl_v2/step3_reward_status.md`.

Full diagnosis: `docs/offline_rl/m4/offline_rl_status.md` and
`docs/offline_rl/m4/offline_rl_m4_iql_status_report.md`.

## 4. Why v2 (clean restart)

We are not patching v1. We are restarting with three corrections:

1. **Fix the supervision target.** The new collector writes a single
   `action` column that is the *actual* action taken by the behaviour
   policy each step, with verified non-zero variance. The collector
   **fails fast** if any controlled action column has near-zero
   variance — the M2 silent-degeneracy mode is a hard error in v2.
   *(In v2 step 2 this guard immediately caught a second bug — see
   §3 item 5 — vindicating the design.)*
2. **Fix the behaviour policy.** v2 uses `RuleBasedPolicyV2`
   (`algorithms/offline_rl_v2/rbc_v2.py`), which subclasses the v1 RBC
   and remaps the broken bare-name obs lookups to the env's
   charger-namespaced names. The RBC's *intended* logic (PV bonus,
   emergency, deadline-aware charging) now actually executes.
3. **Fix the reward.** v2 uses a **KPI-aligned reward** that includes
   *every* dimension the agent's actions influence (cost, carbon, peak,
   ramping, unserved energy). Weights are calibrated against episode-
   level KPI deltas from RBC rollouts so the per-step reward correlates
   with end-of-episode KPI improvement.
4. **Fix the data volume.** v2 collects **N = 10 RBC rollouts** with
   different seeds → ≈ 87 600 transitions, vs the single 8 760-row
   rollout in v1.

The dataset **schema is fixed across behaviour-policy swaps** — only
the values in `action` and `reward` change when we replace the
behaviour policy. This makes the pipeline reusable.

## 5. Working agreement

- **Isolation.** All v2 work lives in `*_v2/` siblings of v1 trees.
  No file outside these paths is modified. No new git branch.
  - `datasets/offline_rl_v2/`
  - `docs/offline_rl_v2/`
  - `algorithms/offline_rl_v2/`
  - `scripts/*_v2.py`
  - `runs/offline_bc_v2/`, `runs/offline_iql_v2/`
  - `tests/offline_rl_v2/`
- **Storage.** Datasets are **Parquet** (typed, compressed). One small
  CSV sample (≤ 1 000 rows) is committed per dataset for human
  inspection.
- **Single building.** Building 5 only, until BC-v2 and IQL-v2 both beat
  RBC on the KPIs.
- **No-touch v1.** v1 code, datasets, runs and docs are never modified.
- **Plan-mode default.** Non-trivial work is planned and reviewed before
  any file is written.
- **Tests before merge.** `pytest tests/offline_rl_v2/` must pass.
- **Tone.** Supervisor-facing. Plain English first, technical detail in
  subsections. No marketing adjectives.

## 6. Pipeline (one diagram)

```
  ┌──────────┐   collect    ┌────────────────────┐   train    ┌──────────────┐
  │   RBC    │ ───────────▶ │ dataset_v2 (RBC)   │ ─────────▶ │  BC-v2       │
  └──────────┘              └────────────────────┘            └──────────────┘
                                       │                              │
                                       │   train                      │
                                       └─────────────────▶ IQL-v2 ◀───┘
                                                          │
                                          if IQL-v2 > RBC │
                                                          ▼
                              ┌────────────────────────────────────┐
                              │ swap behaviour policy (RBC → IQL)  │
                              │ collect dataset_v2.1               │
                              │ retrain BC-v2.1, IQL-v2.1          │
                              └────────────────────────────────────┘
```

The reward function is **fixed once** (calibrated on RBC rollouts) and
re-used across iterations, so improvements from one generation to the
next are measured on the same yardstick.
